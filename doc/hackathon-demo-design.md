# Praxis ハッカソンデモ版 設計書（1ヶ月スコープ）

[フル設計書](motor-assist-app-design.md) をハッカソンデモ用に絞り込んだ版。
**目的はプロダクトの完成ではなく、「3分のデモで審査員に刺さる縦串の実証」**。判断に迷ったら常にこの一文に戻る。

- 対象: Snapdragon X 搭載 Copilot+ PC（開発機 = デモ機 1台に固定）
- AI基盤: Windows AI Foundry（Windows ML + QNN EP / Phi Silica）
- 期限: 1ヶ月（週次マイルストーン制）

---

## 1. デモコンセプト

> **「手で、顔で、声で。—— 体の状態が変わっても、同じPCをずっと使い続けられる」**

3つの入力チャネルが **Intent Bus（入力抽象化層）** の上で同じ操作にマッピングされることを、入力手段を段階的に「失っていく」1本のストーリーで見せる。機能を3つ並べるのではなく、**アーキテクチャの縦串がそのままデモの演出になる**構成。

### 採用する3機能

| # | 機能 | デモでの役割 |
|---|---|---|
| F1 | **ハンドサイン操作** | 掴み。動きが派手で客席から見える |
| F2 | **頭部ポインティング + まばたきクリック** | 本題。「手を一切使っていない」が一目瞭然 |
| F3 | **音声エージェント操作**（Phi Silica） | クライマックス。AI Foundry を使う意味の証明 |

### 明示的にやらないこと（スコープ外リスト）

フル設計から以下を**意図的に落とす**。デモ後に議論になったら「設計済み・ロードマップにある」と答えられるようフル設計書を添える。

- HID フィルタドライバ / 仮想 HID / uiAccess 署名（注入は `SendInput` のみ。デモアプリは通常権限で動くものに限定）
- 視線推定・スキャンモード・意図予測パネル・震え補正（※震え補正は §7 のストレッチ枠）
- 個人化学習・キャリブレーションの永続化（デモ直前に手動キャリブレーションで足りる）
- 暗号化保存・テレメトリ・インストーラ・複数マシン対応・Arm 以外の EP

## 2. デモシナリオ（3分想定）

| 時間 | シーン | 操作 | 見せ方 |
|---|---|---|---|
| 0:00 | 導入 | — | 「日本には上肢に障害のある人が◯万人。専用機器は数十万円。このPCにはNPUがある」 |
| 0:20 | **F1 ハンドサイン** | ✋パーでポインタ追従 → ✊グーで掴んでウィンドウ移動 → 👌でクリック → スワイプでスライド送り | 画面隅にカメラHUD（手のランドマーク描画）。ジェスチャー認識のラベルを大きく表示 |
| 1:00 | 転換①「事故や病気で、手が動かなくなったら？」 | **手を下ろす** | 一拍おく。ここが感情の起点 |
| 1:10 | **F2 頭部+まばたき** | 頭の向きでポインタ → 2秒注視（dwell）または強いまばたきでクリック → F1と同じ操作を続行 | 「同じ操作が、入力を乗り換えて続く」ことを口頭で強調（= Intent Bus） |
| 1:50 | 転換②「さらに、首も動かせなくなったら？」 | — | — |
| 2:00 | **F3 音声エージェント** | 「メモ帳を開いて」「『来週の打ち合わせは木曜に変更でお願いします』と入力」「保存して閉じて」 | 認識テキスト→解釈された意図→実行される操作、の3段をHUDに流す |
| 2:30 | **Wi-Fi 切断の儀** | 機内モードON → F1〜F3をもう一周高速で | 「すべてこの端末のNPUの中。クラウド接続ゼロ、API課金ゼロ、映像は1バイトも外に出ない」 |
| 2:50 | 締め | — | タスクマネージャーのNPUグラフを見せて「CPUもGPUも空いている。だから一日中動かせる」 |

### 演出装置（機能と同格の優先度で作る）

- **HUD オーバーレイ**: カメラ小窓 + ランドマーク描画 + 現在のアクティブチャネル表示 + 認識ジェスチャー/音声のラベル。「AIが見ている様子」の可視化はデモの生命線
- **NPU 使用率の常時表示**: タスクマネージャーを画面端に常駐（Snapdragon である必然性の証明）
- **機内モードスイッチ**: 完全ローカルの証明。ローカルAIデモで最も歓声が上がる瞬間なので必ず入れる

## 3. デモ版アーキテクチャ

フル設計の L0〜L5 を、単一プロセス + 最小限の分離に畳む。

```
┌──────────────────────────────────────────────────────┐
│ HUD/設定 (WinUI 3)  カメラ小窓・ランドマーク・状態表示   │
├──────────────────────────────────────────────────────┤
│ 意味層: Phi Silica (Windows App SDK)                  │
│   音声テキスト → {open_app|type_text|click|save|...}  │
│   ルールベース意図解釈をフォールバックに併設(§6)        │
├──────────────────────────────────────────────────────┤
│ Intent Bus (縦串・デモの主役)                          │
│   全チャネルを5種の抽象イベントに正規化:                │
│   Point / Select / Grab / Text / Command              │
│   チャネル切替は手動ホットキー+自動(顔のみ検出で頭部へ)  │
├──────────────────────────────────────────────────────┤
│ 知覚層: Windows ML + QNN EP @ Hexagon NPU             │
│   ・手: ハンドランドマーク 21点 → ジェスチャーFSM       │
│   ・顔: フェイスランドマーク → 頭部姿勢 + まばたき検出   │
│   ・音声: VAD + Whisper系ASR (ONNX/QNN最適化版)        │
├──────────────────────────────────────────────────────┤
│ センサ/出力: MediaFoundation(カメラ) / WASAPI(マイク)   │
│   注入: SendInput のみ / アプリ操作: UIA + ショートカット │
└──────────────────────────────────────────────────────┘
```

### 技術選定（デモ最短ルート）

| 要素 | 選定 | 理由 |
|---|---|---|
| 知覚モデル | **Qualcomm AI Hub の事前最適化モデル**（MediaPipe Hand / Face 系のランドマーク検出）を第一候補に。なければ公開 ONNX を INT8 量子化 | 自前学習ゼロ。NPU 最適化済みを持ってくるのが1ヶ月の生命線 |
| 推論ランタイム | Windows ML（ONNX Runtime + QNN EP） | AI Foundry の看板。EP 自動選択で「CPUフォールバックでも動く」保険付き |
| ASR | Whisper 小型モデル（QNN 最適化版）。間に合わなければ Windows 標準音声認識 | デモの発話は日本語10フレーズ程度 → 小型で足りる |
| 意図解釈 | **Phi Silica**（Windows App SDK の Language Model API） | Foundry Local の大型LLMはデモの操作範囲では過剰。OS内蔵で配布も楽 |
| ジェスチャー認識 | ランドマーク → ルールベース/軽量FSM（学習しない） | 5ジェスチャーなら指の開閉角のルールで十分。学習パイプラインは作らない |
| アプリ実装 | WinUI 3 (C#) + P/Invoke（SendInput, Raw Input） | Arm ネイティブ・オーバーレイ描画・開発速度のバランス |
| 操作対象 | PowerPoint・メモ帳・エクスプローラーの3つに限定 | UIA 対応が素直で、審査員が知っているアプリ |

### ジェスチャーセット（誤認識耐性を優先して5個に限定）

| ジェスチャー | 割当 | 選定理由 |
|---|---|---|
| ✋ パー(手を開く) | ポインタ追従 ON | ランドマーク的に最も安定 |
| ✊ グー | 掴む(ドラッグ) | パーとの識別が容易 |
| 👌 OKサイン | クリック | 親指-人差し指距離で頑健に取れる |
| 👉→ 水平スワイプ | スライド送り/戻し | 動きが客席から見えて映える |
| 🤙 シャカ(親指+小指) | チャネル切替/メニュー | 他と混同しにくい形状 |

※ ピースサインや複雑な形は照明・角度で事故りやすいので入れない。**「確実に動く5個」>「たまに動く10個」**。

## 4. 週次マイルストーン

各週の終わりに「その時点でデモできる状態」を維持する（垂直スライス開発）。最悪どの週で止まってもデモは成立させる。

### Week 1: パイプライン貫通（技術リスクを最初に潰す週）

- [ ] 開発機で Windows ML + QNN EP のサンプル推論を実行、**NPU で動いていることをタスクマネージャーで確認**
- [ ] ハンドランドマークモデルを NPU で 30fps 実行（Qualcomm AI Hub モデルの動作確認）
- [ ] カメラ取得 → 推論 → ランドマークを画面描画（HUD の原型）
- [ ] パー/グーの2ジェスチャー → SendInput でポインタ移動・クリックの貫通デモ
- **Exit条件: 「手でポインタが動く」動画が撮れる。QNN で動かない演算子等の問題はこの週に洗い出す**

### Week 2: F1完成 + F2実装

- [ ] ジェスチャー5種の FSM + 誤認識対策（数フレーム連続で確定、確定時に効果音）
- [ ] Intent Bus 導入（Point/Select/Grab/Text/Command への正規化、チャネル切替）
- [ ] フェイスランドマーク → 頭部姿勢ポインティング（感度カーブ調整）+ まばたき/dwell クリック
- [ ] PowerPoint・エクスプローラーでの操作シナリオ確認
- **Exit条件: F1→F2 の乗り換えデモが通しで動く**

### Week 3: F3（音声）+ 統合

- [ ] VAD + ASR を NPU で常時実行（デモ用10フレーズの認識率を優先チューニング）
- [ ] Phi Silica で意図解釈 → コマンドスキーマ（open_app / type_text / click_element / save / close）
- [ ] ルールベースフォールバック実装（§6）
- [ ] UIA でメモ帳への入力・保存を実行
- [ ] HUD 完成（チャネル表示・認識ラベル・音声3段表示）
- **Exit条件: 3分シナリオが最後まで一度は通る**

### Week 4: デモ品質への引き上げ（機能追加禁止週）

- [ ] シナリオ通し練習 20本以上。失敗パターンの記録と対策
- [ ] 会場想定の照明・立ち位置・カメラ角度でのテスト（逆光・スポットライトが最大の敵）
- [ ] カメラ露出固定・ホワイトバランス固定の設定 UI
- [ ] 失敗時のリカバリ動線整備（§6）と機内モード演出のリハーサル
- [ ] 発表資料・ピッチ練習（フル設計書を「今後の構想」として添付）
- **Exit条件: 誰が操作しても8割成功する / 発表者なら10割**

## 5. デモ成立条件（数値目標）

プロダクト品質ではなくデモ品質で定義する。

| 指標 | 目標 | 備考 |
|---|---|---|
| ジェスチャー認識成功率 | 発表者による実演で 95%以上 | 発表者の手・顔でチューニングしてよい（汎化は捨てる） |
| ポインタ遅延（体感） | 100ms 以下 | フル設計の45msは追わない。「遅れて見えない」でOK |
| 音声コマンド成功率 | デモ用10フレーズで 95%以上 | 自由発話の頑健性は捨て、シナリオのフレーズに過学習させる |
| 連続稼働 | 15分間クラッシュなし | 発表前の待機時間を含めて耐える |
| 全処理のオフライン動作 | 機内モードで全機能動作 | Phi Silica・ASR・知覚すべて（これが崩れると訴求の根幹が崩れる） |

## 6. デモ事故対策（リスク&フォールバック）

| リスク | 対策 |
|---|---|
| 会場照明でランドマーク検出が不安定 | 露出・WB固定機能 / 前日に会場でカメラ位置決め / 最悪は手元照明（リングライト）を持参 |
| ジェスチャー誤発火 | 確定に N フレーム連続一致 + 確定音。デモ中は使わないジェスチャーを無効化する「シナリオモード」 |
| Phi Silica が応答しない/遅い | **ルールベース意図解釈を常時併設**（デモ10フレーズはキーワードマッチで100%解釈可能にしておく）。HUD表示は同一なので観客には区別がつかない |
| ASR がマイク環境で崩れる | ピンマイク（有線）持参 / デモフレーズを認識しやすい語彙に事前調整 |
| QNN EP で特定モデルが動かない | Week 1 で検証済みにする。最悪 CPU フォールバック（Snapdragon 訴求は NPU グラフの代わりに「オフライン」に切替） |
| SendInput が対象アプリに効かない | 操作対象3アプリで事前検証済みに。昇格が必要なアプリはシナリオに入れない |
| 本番でのフリーズ | ホットキー1発で全状態リセット（プロセス再起動 3秒以内）。デモ動画をバックアップとして必ず用意 |

## 7. ストレッチゴール（本命が Week 3 で完成した場合のみ）

1. **震え補正 before/after**: 低レベルフック（`WH_MOUSE_LL`）版の簡易フィルタで、カーソル軌跡の比較画面を追加。「意図的に手を震わせる」演出の是非はチームで判断（当事者性の観点で §フル設計 7 参照）
2. **スライド送りの Bluetooth 連携**: 別画面（プロジェクター）の PowerPoint をハンドサインで遠隔操作 — ステージの見栄えが上がる
3. **HUD の英語切替**: 審査員に海外メンバーがいる場合用

## 8. ピッチでの位置づけ（技術デモを事業ストーリーに接続する一言）

- 「今日見せたのは3チャネルだが、**Intent Bus の上にはあと4チャネル載る**（視線・スイッチ・既存デバイス補正・意図予測）」→ フル設計書を提示
- 「専用視線入力デバイスは10万円超。**これは Copilot+ PC を買えば追加費用ゼロ**」
- 「クラウドAIでは常時稼働の電力も、映像を送るプライバシーも成立しない。**NPU搭載PCでしか作れないアプリ**」
