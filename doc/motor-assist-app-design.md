# 運動機能支援アプリ設計書（仮称: Praxis）

Snapdragon X シリーズ搭載 Copilot+ PC を対象に、**専用機器なし（内蔵カメラ・マイク・NPUのみ）で PC を完全操作可能にする**運動機能支援アプリの設計。
Microsoft **Windows AI Foundry** をAI実行基盤とし、入力スタックの**低レイヤ（HID/入力注入層）まで統合**することを設計の核とする。

> 出典: [local-ai-features-focus.md](local-ai-features-focus.md) B-3「運動機能支援」の発展形

---

## 1. コンセプトと対象ユーザー

### 1.1 コンセプト

> 「手が使えなくても、手が震えても、**追加ハードウェアなし・月額なし・データ送信なし**で Windows のすべてを操作できる」

- 従来の視線入力デバイス（Tobii 等）は数万〜数十万円の専用機器が必要。内蔵カメラ + Hexagon NPU で代替する
- 支援機能は「常時稼働」が絶対条件 → 低電力 NPU でバッテリー駆動でも一日中動くことが成立条件
- 操作データ・映像・障害情報という最高機密の個人データを 1 バイトも端末外に出さない

### 1.2 対象ユーザー（ペルソナ）

| ペルソナ | 状態 | 主に使う機能 |
|---|---|---|
| P1: 振戦のあるユーザー | パーキンソン病・本態性振戦 | 震え補正フィルタ（既存マウスを使い続ける） |
| P2: 上肢の麻痺・欠損 | 頸髄損傷・脳性麻痺・欠損 | 頭部ポインティング + 表情スイッチ + 音声 |
| P3: 進行性疾患 | ALS・筋ジストロフィー | 視線 + わずかな表情 + 意図予測（進行に応じ入力チャネルを段階的に切替） |
| P4: 一時的な障害 | 骨折・腱鞘炎 | 音声完全操作（一般ユーザーへの裾野拡大の入口） |

進行性疾患（P3）では「残存機能の変化に合わせて入力チャネルを乗り換えられる」ことが決定的に重要 → **複数入力チャネルを同一の抽象化層に載せる**設計とする（§4）。

## 2. 機能設計

### 2.1 入力チャネル（センシング → 操作イベント）

| # | チャネル | 内容 | 使用ハードウェア |
|---|---|---|---|
| IN-1 | 頭部ポインティング | 頭部姿勢（pitch/yaw）でポインタを相対/絶対制御 | カメラ + NPU |
| IN-2 | 視線推定 | 注視領域の推定（頭部と融合して精度確保） | カメラ + NPU |
| IN-3 | 表情スイッチ | まばたき（片目/両目/長閉眼）、眉上げ、口の開閉、頬の動き → クリック/修飾キー | カメラ + NPU |
| IN-4 | ハンドジェスチャー | 可動域が限られる手の小さな動きをジェスチャー化 | カメラ + NPU |
| IN-5 | 音声 | 常時ウェイクワードレス音声コマンド + 自由発話エージェント + ディクテーション | マイク + NPU |
| IN-6 | 既存デバイス補正 | 物理マウス/タッチパッド/ジョイスティックの震え・不随意動作をフィルタ | HID 低レイヤ（§5） |
| IN-7 | スイッチデバイス | 市販の1〜2接点スイッチ（既存AAC資産）をスキャン操作に接続 | USB HID |

### 2.2 操作・出力機能

- **OUT-1 ポインタ制御**: 速度カーブ・デッドゾーン・慣性をユーザーごとに学習調整
- **OUT-2 クリック方式の選択**: dwell（滞留）クリック / 表情スイッチ / 音声「クリック」/ 物理スイッチ — 任意の組み合わせ
- **OUT-3 スキャンモード**: ポインティング不能なユーザー向けに、UI 要素をハイライト巡回 → スイッチ 1 つで選択
- **OUT-4 セマンティック操作**: 「ピクセル座標」ではなく **UI Automation ツリーの要素**を直接ターゲット（「保存ボタン」を要素として押す）。小さいターゲットへの精密ポインティング自体を不要化する
- **OUT-5 仮想キーボード + 予測入力**: Phi Silica による文脈補完で 1 文字ずつの入力回数を削減。定型文・語彙のローカル学習
- **OUT-6 音声エージェント操作**: 「請求書の PDF を開いて 2 ページ目を印刷して」→ LLM がプラン化 → UIA 操作列として実行（確認 UI を挟む）
- **OUT-7 意図予測パネル**: 現在の画面・直前の操作・時刻から「次にやりそうな操作」上位 3〜5 件を大きなボタンで常時提示。操作回数を 1/10 にする本命機能
- **OUT-8 ゲーム/フルスクリーンモード**: 仮想 HID（§5.2）経由で注入するため、Raw Input 直読みのゲームでも頭部操作・震え補正が効く

### 2.3 個人化・適応

- **振戦プロファイル学習**: 個人の震えの周波数・振幅特性をローカル学習し、固定フィルタより高精度に意図成分を抽出
- **キャリブレーションの継続更新**: 視線・頭部のキャリブレーションを使用中に暗黙的に補正（クリック成功位置を教師信号に）
- **疲労適応**: 操作精度の低下を検知したら dwell 時間・ターゲット拡大率を自動で緩める
- **進行対応プリセット**: P3 向けに「今日は調子が悪い」ワンタップで入力チャネル構成を切替

## 3. プラットフォームと Windows AI Foundry の使い分け

### 3.1 対象デバイス

- **Snapdragon X Elite / X Plus 搭載 Copilot+ PC**（Windows 11 on Arm）
- Hexagon NPU（45 TOPS）を知覚パイプラインの主実行体とし、CPU/GPU は制御・UI・バースト推論に温存

### 3.2 Windows AI Foundry コンポーネントのマッピング

Windows AI Foundry は「Windows ML(基盤ランタイム) + Windows AI APIs(内蔵モデル) + Foundry Local(モデルカタログ)」の3層。本アプリでは以下のように使い分ける。

| レイヤ | 使用コンポーネント | 本アプリでの用途 |
|---|---|---|
| カスタム視覚モデル | **Windows ML**（ONNX Runtime + **QNN Execution Provider** で Hexagon NPU 実行） | 顔ランドマーク / 頭部姿勢 / 視線推定 / 手指ポーズ / 表情分類。INT8 量子化 ONNX を同梱し、Windows ML の EP 自動選択で NPU に載せる（AMD/Intel NPU への将来展開も同じバイナリで可能） |
| 内蔵 AI API | **Windows AI APIs**（Windows App SDK） | **Phi Silica**: 音声コマンドの意図解釈・予測入力・意図予測パネル / **Text Recognition (OCR)**: UIA が取れないレガシーアプリ・画像化 UI の画面理解フォールバック |
| 大きめ LLM | **Foundry Local** | 音声エージェントの複雑なタスクプラン（複数手順の操作列生成）、UIA ツリー + 履歴からの高度な意図予測。NPU 最適化済みモデルをカタログから取得・OTA 更新 |
| 音声認識 | Windows ML 上の Whisper 系 ONNX（QNN 最適化版） | 常時 VAD + ストリーミング ASR。OS 標準音声認識より語彙適応・低遅延を優先し自前実行 |

**設計判断**: 知覚（視覚・音声）は「自前 ONNX + Windows ML/QNN EP」、言語理解は「Phi Silica を第一選択、重いタスクのみ Foundry Local」。理由: 知覚モデルは 30fps 常時実行のため量子化・演算子の完全なコントロールが必要。言語系は OS 内蔵モデル（Phi Silica）ならストレージ・メモリ・更新コストをOSに肩代わりさせられる。

## 4. システムアーキテクチャ

```
┌────────────────────────────────────────────────────────────────┐
│ L5 UX層          設定/キャリブレーションUI・オーバーレイHUD        │
│                  （意図予測パネル・スキャンハイライト・仮想KB）      │
├────────────────────────────────────────────────────────────────┤
│ L4 意味層        Phi Silica / Foundry Local                     │
│  (バースト)      ・音声→意図→操作プラン  ・意図予測  ・予測入力     │
│                  ・UI Automation ツリー理解 + OCRフォールバック    │
├────────────────────────────────────────────────────────────────┤
│ L3 制御層 (CPU)  入力抽象化バス「Intent Bus」                     │
│                  ・震えフィルタ(Kalman+個人学習モデル)             │
│                  ・dwell/スキャン/ジェスチャーFSM ・速度カーブ      │
│                  ・チャネル合成(頭部×視線×表情の融合)              │
├────────────────────────────────────────────────────────────────┤
│ L2 知覚層 (NPU常時) Windows ML + QNN EP @ Hexagon               │
│                  ・顔/頭部/視線/手指/表情 30fps  ・VAD/ASR        │
├────────────────────────────────────────────────────────────────┤
│ L1 センサ層      MediaFoundation(カメラFrame Server共有)         │
│                  WASAPI(マイク)  Raw Input(物理HID)              │
├────────────────────────────────────────────────────────────────┤
│ L0 OS統合層(低レイヤ) ※§5                                       │
│                  ・仮想HIDデバイス(UMDF2/VHF)による入力注入        │
│                  ・HIDフィルタによる物理入力の横取り(震え補正)      │
│                  ・SendInput+uiAccess  ・UI Automation Client    │
└────────────────────────────────────────────────────────────────┘
```

### 4.1 Intent Bus（入力抽象化バス）— 設計の要

すべての入力チャネル（IN-1〜7）は生データのまま使わず、**「ポインティング」「選択」「修飾」「テキスト」「コマンド」の 5 種の抽象イベント**に正規化して Intent Bus に流す。

- 出力側（OUT-1〜8）はどのチャネル由来かを知らない → **チャネルの追加・乗り換え・組み合わせが自由**（P3 の進行対応の実現手段）
- 例: 「ポインティング=頭部、選択=まばたき」「ポインティング=視線、選択=物理スイッチ」「全部=音声」が同じ配線で成立
- 各イベントには信頼度スコアを付与し、L4 の意図予測が低信頼イベントの曖昧性解消を行う（「ボタン A と B の中間で dwell → 文脈的に A だろう」）

### 4.2 プロセス構成

| プロセス | 権限 | 役割 |
|---|---|---|
| Praxis Perception Service | 通常（Windows サービス） | L1-L2。カメラ/マイク→NPU 推論→ランドマーク座標のみを共有メモリへ（映像フレームはこのプロセスから出ない） |
| Praxis Core | uiAccess=true（署名+Program Files 配置） | L3-L4。Intent Bus、注入、UIA 操作、LLM 呼び出し |
| Praxis HID Driver | カーネル/UMDF2 | L0。仮想 HID 出力 + 物理 HID フィルタ（§5） |
| Praxis UI | 通常 | L5。設定・オーバーレイ描画（DirectComposition 最前面） |

## 5. 低レイヤ設計（本アプリの技術的護城河）

「アプリ層の擬似操作」で済ませず入力スタックの下に入ることが、競合の Web/アプリ層ソリューションとの決定的な差になる。

### 5.1 なぜ低レイヤが必要か

1. **震え補正（IN-6）は入力の発生源で行う必要がある**: アプリが Raw Input やポーリングで直接デバイスを読む場合（ゲーム・CAD・リモートデスクトップ）、アプリ層のフックでは補正が届かない
2. **注入した操作が「本物の入力」として扱われる必要がある**: `SendInput` は UIPI に阻まれ、管理者昇格ウィンドウ・UAC セキュアデスクトップ・アンチチートを持つゲームに届かない/弾かれる
3. **支援技術はログオン画面・UAC ダイアログでも動かなければならない**: ここで操作不能になる支援ツールは「一人で PC を使える」を達成できない

### 5.2 入力注入: 仮想 HID デバイス方式

- **UMDF2 + Virtual HID Framework (VhfCreateDevice)** で仮想マウス/キーボード/ゲームパッドをデバイスノードとして生成
- OS からは物理デバイスと不可分 → Raw Input・DirectInput・昇格プロセス・ログオン画面すべてに入力が届く
- 併用: 通常デスクトップの軽量な操作は `SendInput`（低コスト）、フルスクリーン/昇格先は仮想 HID、と自動切替
- `uiAccess=true` マニフェスト + EV コード署名 + `Program Files` 配置で、UIA 経由のセキュアな UI 操作も確保（Windows の支援技術向け正規ルート）

### 5.3 物理入力のフィルタリング（震え補正の実装）

- **HID クラスの upper filter ドライバ**として物理マウス/パッドの HID レポートを横取り → L3 のフィルタ（個人学習済み振戦モデル + Kalman）を通した座標を**仮想 HID から再送出**、元のレポートは破棄
- これにより「アプリから見ると最初から震えていないマウス」になる。全アプリ・全ゲーム・リモート先にも一律で効く
- フィルタモデルの実行はカーネルでなくユーザーモード（L3）で行い、ドライバは転送のみ（クラッシュ耐性・更新容易性）
- フェイルセーフ: サービス死活監視で、Core 停止時はドライバが自動パススルーに戻る（**支援ツール自身が入力を奪って操作不能にする事故を構造的に防ぐ**）

### 5.4 画面・UI への低レイヤアクセス

- **UI Automation (UIA) クライアント**として全アプリの UI ツリーを取得 → OUT-4/OUT-7 のターゲット列挙
- UIA 非対応のレガシー/画像化 UI は、Windows AI APIs の **OCR + セグメンテーション**でスクリーンショットから擬似 UI ツリーを構築（Snapdragon NPU で数十 ms）
- オーバーレイは DirectComposition のクリック透過トップモストレイヤで描画（ゲームのフルスクリーン時は独立フリップに割り込めないため、ボーダレスウィンドウ推奨を案内）

### 5.5 カメラ・電力の低レイヤ配慮

- カメラは Windows の **Frame Server 経由で共有取得**（Windows Studio Effects やビデオ会議と同時使用可能に）。Praxis 用には低解像度ストリーム（640×480）を要求し帯域と電力を節約
- **動的フレームレート**: 操作アイドル時は 5fps の存在検知のみ → 顔が動き始めたら 30fps に昇格（NPU 消費を最小化）
- Modern Standby 復帰・ロック画面でも Perception Service を維持（サービス + セッション 0 分離を考慮した構成）

## 6. レイテンシ・電力予算

### 6.1 レイテンシ予算（ポインティング end-to-end 目標 < 50ms）

| 区間 | 予算 | 実現手段 |
|---|---|---|
| カメラ露光〜フレーム取得 | ~33ms (30fps) | 60fps モードで 16ms に短縮するオプション |
| 前処理（クロップ・正規化） | 2ms | CPU (Arm NEON) |
| NPU 推論（顔+視線+表情） | 5–8ms | QNN EP・INT8・モデル連結で 1 パス化 |
| フィルタ・マッピング | 1ms | L3 |
| 注入（仮想 HID/SendInput） | 1ms | L0 |
| **合計** | **~45ms** | ポインタ追従の体感目標（遅延感知閾値 ~60-70ms 未満） |

- 音声コマンド: VAD 検知〜Phi Silica 意図解釈〜実行 < 800ms 目標
- 音声エージェント（Foundry Local の複数手順プラン）: < 3s + 実行中の逐次フィードバック

### 6.2 電力予算

| 状態 | 稼働内容 | 目標消費 |
|---|---|---|
| アイドル監視 | 5fps 存在検知 + VAD のみ | < 0.3W |
| アクティブ操作 | 30fps 知覚 + フィルタ | < 1.5W |
| エージェント実行 | + Foundry Local LLM バースト | 瞬間 5–8W（数秒） |

→ 一日 8 時間のアクティブ利用でもバッテリー影響を 15% 以下に抑える。**この電力プロファイルは CPU/GPU 実行では成立せず、Hexagon NPU が製品成立の条件**。

## 7. プライバシー・セキュリティ設計

- **映像非保存原則**: カメラフレームは Perception Service 内で即時ランドマーク座標（数百バイト/フレーム）に変換され、フレームバッファはプロセス外に出ない・ディスクに書かれない
- **全処理オンデバイス**: ネットワーク権限は「モデル OTA 更新（Foundry Local カタログ）」のみに限定し、テレメトリはデフォルトオフ・明示オプトイン（送るのは匿名の性能統計のみ）
- **個人化データの保護**: 振戦プロファイル・視線キャリブレーション・語彙は DPAPI で暗号化してローカル保存。エクスポート/削除はワンタップ（本人のデータ主権）
- **注入経路の防御**: 仮想 HID / uiAccess は攻撃価値が高い → Praxis Core との IPC は署名検証 + 名前付きパイプ ACL で第三者プロセスからの注入要求を遮断
- **医療機器非該当の設計**: 診断・治療を目的とせず「操作支援・ウェルネス」に機能を限定（§B-5 系の症状トレンド分析は本アプリには載せない判断）

## 9. リスクと対応

| リスク | 対応 |
|---|---|
| 内蔵カメラのみの視線推定精度（±2–3° が限界） | 視線単独に頼らず「頭部で粗く・視線で補正・UIA ターゲットにスナップ」の融合設計（OUT-4 が精度要求自体を下げる） |
| QNN EP の演算子カバレッジ不足でモデルの一部が CPU フォールバック | モデル設計段階で QNN 対応演算子に限定 / Windows ML の EP 自動選択で世代差を吸収 |
| HID フィルタドライバの署名・配布コスト（WHQL） | MVP では filter なしの Raw Input + low-level hook 実装で先行検証し、ドライバは並行開発 |
| uiAccess の EV 証明書・Store 配布との両立 | Store 版（uiAccess なし・通常デスクトップのみ）と支援技術版（フル機能・直接配布）の 2 SKU |
| アンチチートによる仮想 HID 検出 | ゲームモードは「対応タイトルリスト」方式で段階展開。アンチチートベンダーと支援技術ホワイトリスト登録の交渉 |
| Phi Silica / Foundry Local の API 変化 | L4 の LLM 呼び出しを内部アダプタで抽象化し、モデル/API 差し替え可能に |
